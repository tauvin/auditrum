# Django integration

Everything you need to run auditrum inside a Django project. Assumes
you've skimmed [Core concepts](concepts.md) or at least the
[getting started](getting-started.md) Django path.

## Install

```bash
pip install "auditrum[django]"
```

Then wire the app and middleware:

```python
# settings.py
INSTALLED_APPS = [
    # ...
    "django.contrib.contenttypes",
    "django.contrib.auth",
    "auditrum.integrations.django",
]

MIDDLEWARE = [
    # ...
    "auditrum.integrations.django.middleware.AuditrumMiddleware",
    # ...
]
```

`AuditrumMiddleware` registers a `connection.execute_wrapper` for the
lifetime of the request. See
[Core concepts → context propagation](concepts.md#context-propagation)
for the mechanics.

## Marking models for tracking

The canonical API is the `@track` decorator:

```python
# myapp/models.py
from django.db import models
from auditrum.integrations.django import track, AuditedModelMixin


@track(fields=["status", "total"])
class Order(AuditedModelMixin, models.Model):
    status = models.CharField(max_length=32)
    total = models.DecimalField(max_digits=10, decimal_places=2)
    created_at = models.DateTimeField(auto_now_add=True)
```

`fields=[...]` is the whitelist; pass `exclude=[...]` instead for a
blacklist. You can't pass both — the decorator raises at decoration
time.

`AuditedModelMixin` is optional. It adds instance-level helpers
(`order.audit_events`, `order.audit_at(ts)`, `order.audit_field_history('status')`)
and a classmethod `Order.audit_state_as_of(ts)`. None of them touch the
schema — everything routes through the shared `auditlog` table.

### All decorator arguments

| Argument         | Type             | Purpose                                                                             |
|------------------|------------------|-------------------------------------------------------------------------------------|
| `fields`         | `list[str]`      | Whitelist of columns to include in diff / old_data / new_data                       |
| `exclude`        | `list[str]`      | Blacklist; mutually exclusive with `fields`                                         |
| `extra_meta`     | `list[str]`      | `NEW.<field>` references captured per-row into `auditlog.meta` jsonb                |
| `log_condition`  | `str`            | PL/pgSQL expression; trigger is a no-op when false. **Trusted input only**          |
| `audit_table`    | `str`            | Override the destination audit log table name (defaults to project setting)         |
| `trigger_name`   | `str`            | Override default `audit_<table>_trigger` name                                       |

### Legacy `register()` function

If you used 0.2.0 style per-app `audit.py` modules:

```python
# myapp/audit.py
from auditrum.integrations.django import register
from .models import Order

register(Order, track_only=["status"])
```

This still works — it delegates to `@track` internally. The registry
is shared, so `auditrum_makemigrations` picks specs up the same way.

## Migrations

Triggers are real Django migration operations in 0.3+. Generate them
with a dedicated management command:

```bash
# One-time: create auditlog, audit_context, helper functions, partitions
python manage.py migrate auditrum_django

# Generate migration files for @track'd models (run after adding/changing a spec)
python manage.py auditrum_makemigrations

# Preview without writing files
python manage.py auditrum_makemigrations --dry-run

# Apply
python manage.py migrate
```

The command walks the `@track` registry, groups specs by the tracked
model's `app_label`, and writes one migration file into `<app>/migrations/`.
Dependencies are filled in automatically: every generated migration
depends on `auditrum_django.0001_initial` and the latest existing
migration in the target app.

A generated file looks like this:

```python
# myapp/migrations/0013_auditrum_20260414_1108.py
# Auto-generated by auditrum_makemigrations. Review and edit carefully.
from django.db import migrations

from auditrum.integrations.django.operations import InstallTrigger


class Migration(migrations.Migration):
    dependencies = [
        ("auditrum_django", "0001_initial"),
        ("myapp", "0012_auto_rename_column"),
    ]

    operations = [
        InstallTrigger(
            table="myapp_order",
            fields_kind="only",
            fields=["status", "total"],
        ),
    ]
```

`InstallTrigger` / `UninstallTrigger` are real `Operation` subclasses,
so `sqlmigrate`, `showmigrations`, and rollbacks (`database_backwards`)
all work.

### Workflow for schema changes

1. Update model (add a tracked field, change `fields=[...]`).
2. Run `python manage.py auditrum_makemigrations`.
3. Review the generated migration in your PR.
4. Apply with `migrate`.

`TriggerManager.install(force=True)` is what runs inside
`database_forwards`, so re-applying a migration updates the trigger in
place via drift detection. You don't need to manually drop the old
trigger.

## Middleware

`AuditrumMiddleware` does three things on each request:

1. Generates a stable `request.request_id` if one isn't set
2. Collects `{user_id, username, client_ip, user_agent, session_key, request_id, url, method, source: "http"}`
3. Enters an `auditrum_context(**metadata)` block around `get_response(request)`

Inside that block, the Django connection wrapper prefixes every
`cursor.execute` with `set_config` calls that populate the session GUCs
which audit triggers read.

### Limiting which methods trigger context

By default, middleware runs for all HTTP methods. To skip idempotent
`GET` / `HEAD` requests:

```python
# settings.py
PGAUDIT_MIDDLEWARE_METHODS = ("POST", "PUT", "PATCH", "DELETE")
```

### Custom metadata

Subclass to inject your own fields:

```python
# myproject/middleware.py
from auditrum.integrations.django.middleware import AuditrumMiddleware


class TenantAwareAuditrumMiddleware(AuditrumMiddleware):
    def get_context(self, request):
        ctx = super().get_context(request)
        ctx["tenant_id"] = getattr(request, "tenant_id", None)
        ctx["feature_flags"] = list(getattr(request, "feature_flags", []))
        return ctx
```

## ORM helpers

The Django `AuditLog` model ships with `AuditLogManager` / `AuditLogQuerySet`
backed by the `(table_name, object_id, changed_at DESC)` composite index:

```python
from auditrum.integrations.django.models import AuditLog

AuditLog.objects.for_model(Order).recent(50)
AuditLog.objects.for_object(order)
AuditLog.objects.for_user(request.user).order_by("-changed_at")[:20]
AuditLog.objects.for_user(42)                     # or pass a raw user_id
AuditLog.objects.for_user(None)                   # events with no user
AuditLog.objects.for_context(request_context_id)  # every event in one request
AuditLog.objects.by_table("orders")               # raw table name; useful for cross-service audit
```

All methods return a chainable queryset, so you can combine freely:

```python
(
    AuditLog.objects
    .for_user(request.user)
    .for_model(Order)
    .filter(changed_at__gte=yesterday)
    .order_by("-changed_at")
)
```

## `AuditedModelMixin`

Optional mixin on your tracked models. Adds:

```python
order = Order.objects.get(pk=1)

# Every event on this instance
order.audit_events                          # AuditLogQuerySet

# Snapshot of this row's state at a past moment (None if it didn't exist)
from datetime import datetime, UTC
snapshot = order.audit_at(datetime(2024, 6, 1, tzinfo=UTC))
if snapshot:
    print(snapshot.status)                  # attribute access
    print(snapshot["total"])                # dict access
    print(snapshot.data)                    # the raw jsonb row
    historical = snapshot.to_model(Order)   # unsaved Order with historical values
                                            # (fields removed from the model since are ignored)

# Timeline of a single field
for ts, status in order.audit_field_history("status"):
    print(ts, status)

# Class-level: every surviving row at a past timestamp
from datetime import datetime, UTC
for row in Order.audit_state_as_of(datetime(2024, 1, 1, tzinfo=UTC)):
    print(row.object_id, row.data["status"])

# Class-level: every event ever for this model
Order.audit_history().filter(operation="DELETE")
```

See [time travel](time-travel.md) for what `audit_at`, `audit_field_history`,
and `audit_state_as_of` do under the hood.

## Admin integration

`AuditLog` and `AuditContext` are registered with Django admin out of
the box (via `auditrum.integrations.django.admin`). The `AuditLog`
change list includes operation / table / object id / user id / context
id columns, filters on operation and table, and a read-only detail view.

Typical use: add an "Audit history" link to the `ModelAdmin` of any
tracked model using `AuditHistoryMixin`:

```python
# myapp/admin.py
from django.contrib import admin
from auditrum.integrations.django.mixins import AuditHistoryMixin
from .models import Order


@admin.register(Order)
class OrderAdmin(AuditHistoryMixin, admin.ModelAdmin):
    list_display = ("id", "status", "total")
```

`AuditHistoryMixin` adds a per-object history view at
`admin:myapp_order_audit_history` showing the full audit timeline for
that row with paginated diffs.

## Tips and gotchas

- **Tracked models must have a `pk`.** Composite primary keys are not
  supported (`object_id` is a single text column in `auditlog`).
- **The middleware is connection-bound.** If your project uses multiple
  databases, the wrapper is installed on the default alias. Opening a
  second connection inside a request requires manually entering an
  `auditrum_context(**metadata)` with that connection's cursor.
- **`log_condition` is trusted input.** Never `f"NEW.state = '{user_input}'"` — use
  a static PL/pgSQL expression that reads `NEW.*` and the result of
  `current_setting('auditrum.context_metadata')::jsonb` if you need
  dynamic gating.
- **Signals still fire.** Audit triggers run inside the database, so
  Django signals (`pre_save`, `post_save`) are unaffected by their
  presence. Bulk operations (`QuerySet.update`, `bulk_create`) that
  bypass signals are still captured, because triggers live at the DB
  level.
